1

  1. Crear el directorio del proyecto
    mkdir academia
  2. Ingresar al directorio del proyecto
    cd academia
  3. Inicializar el repositorio de git vacio
    git init
  4. Verificar el estado del repositorio de git
    git status
  5. Descargar el Framework de Foundation y descomprimir el contenido dentro del directorio del proyecto
  6. Verificar el estado del repositorio de git
    git status
  7. Añadir todos los archivos al "stage area"
    git add --all
  8. Verificar los archivos que se agregaran al "stage area"
    git status
  9. Realizar el commit
    git commit -m "Initial Commit"
  10. Agregar el repositorio remoto "origin"
    git remote add origin https://github.com/[nombre-de-usuario]/academia.git
  11. Publicar los cambios en el repositorio remoto
    git push -u origin master
Cuando se inicializa un repositorio de git se crea dentro del directorio del proyecto correspondiente un directorio llamado .git/ que contiene el repositorio.
Al descromprimir el framework de Foundation en el directorio de trabajo se añaden muchos archivos al "working directory" y para agregarlos todos a la vez al "stage area" utilizamos el flag --all
Todo archivo agregado al "stage area" se convierte en un "tracked file" es decir que ya tiene "historia" en el repositorio, los "untracked files" serán los nuevos que se agreguen

Es muy recomendable antes de empezar a trabajar obtener todos los cambios con el comando pull, el cual realizará un fetch de los archivos y un merge local
Un commit NO corresponde a guardar los cambios en un archivo, debe corresponder a un nuevo "feature" probado y completo lo cual puede implicar uno o mas archivos

15

  1. Obtener los cambios del repositorio remoto
    git pull origin master
  2. Verificar el historial de cambios
    git log --oneline
  3. Crear una etiqueta (milestone) para el proyecto
    git tag -a 1.0.0 -m "Alpha version"
  4. Verificar el listado de etiquetas
    git tag --list
  5. Publicar la etiqueta en el repositorio remoto
    git push --tags
Establecer etiquetas (tag) es una forma muy útil de identificar las versiones del software

17

  1. Obtener los cambios del repositorio remoto de la nueva rama
    git pull origin browser_compatibility
  2. Verificar las ramas del proyecto
    git branch
  3. Cambiar a la nueva rama
    git checkout browser_compatibility
  4. Verificar el historial de cambios
    git log --oneline
  5. Cambiar a la rama principal
    git checkout master
  6. Mezclar las ramas
    git merge browser_compatibility
  7. Verificar el historial de cambios
    git log --oneline
  8. Publicar los cambios en el repositorio remoto
    git push origin master
  9. Crear una nueva etiqueta para el proyecto
    git tag -a 1.1.0 -m "Alpha version with browser compatibility"
  10. Verificar el listado de etiquetas
    git tag --list
  11. Publicar la etiqueta en el repositorio remoto
    git push --tags
El comando branch permite listar todas las ramas del repositorio local
El comando checkout permite pasar a la rama indicada
El comando merge trae todos los "commit" desde la rama indicada a la rama donde se esta actualmente, git realiza de manera automática la resolución de los conflictos si no puede resolver alguno le indica al usuario y este debe resolverlo manualmente y finalmente un nuevo "commit"
